这是一个非常经典且棘手的 Chrome 扩展开发问题。错误信息 `Could not establish connection. Receiving end does not exist.` 直译是“无法建立连接，接收端不存在”。

结合你提供的代码片段，这个问题**并不是你的消息监听逻辑写错了**（你的 `return true` 处理异步响应的逻辑是完全正确的），而是**执行环境或时机**出了问题。

为了给你一个最全面、最深入的解答，我将从**底层原理**、**六大核心原因**、**针对性解决方案**以及**调试策略**四个维度进行详细拆解。

---

### 一、核心原理：为什么会出现这个错误？

在 Chrome 扩展架构中，`chrome.tabs.sendMessage` 的工作机制如下：
1.  **Background (发送方)** 向特定的 `tab.id` 发送消息。
2.  Chrome 浏览器内核会去查找该 Tab 中是否有一个**已激活且已注入**的 Content Script。
3.  Content Script 中必须有一个正在运行的 `chrome.runtime.onMessage.addListener` 监听器。
4.  **如果找不到这个监听器**（因为脚本没加载、页面受限、脚本崩溃、或者 Tab 刚关闭），Chrome 就会立即抛出 `Could not establish connection. Receiving end does not exist.` 错误。

**关键点**：这个错误通常发生在**消息发送的那一微秒**，接收端“不在场”。

---

### 二、深度排查：导致错误的六大核心原因

#### 1. Manifest 配置不匹配（最常见原因）
Content Script 不会自动在所有页面运行，它完全依赖 `manifest.json` 中的 `matches` 规则。

*   **现象**：你在 A 网站测试，但 `manifest.json` 只写了 B 网站的规则，或者规则写错了。
*   **检查点**：
    *   `matches` 是否覆盖了当前 URL？
        *   ❌ 错误：`"matches": ["https://www.example.com/*"]` (漏掉了不带 www 的，或者漏掉了 http)
        *   ✅ 正确：`"matches": ["*://*.example.com/*"]`
    *   是否包含了 `path`？如果你的 URL 是 `example.com/sub/page`，而规则是 `example.com/`，通常是可以的，但如果规则太严格（如 `example.com/index.html`），子页面就无法匹配。
    *   **Manifest V3 特别注意**：确保你有 `host_permissions` 权限，虽然 `content_scripts` 的 matches 会自动申请权限，但在某些动态场景下显式声明更安全。

#### 2. “受限页面”限制 (Restricted Schemes)
Chrome 出于安全考虑，**严禁** Content Script 注入到以下类型的页面。如果你尝试向这些页面的 Tab 发送消息，**100%** 会报这个错：
*   `chrome://` (设置页、扩展管理页等)
*   `chrome-extension://` (扩展自己的页面，除非特殊配置)
*   `about:blank`
*   `about:newtab`
*   `file://` (本地文件，需要在扩展设置中手动开启“允许访问文件 URL”)
*   `https://chrome.google.com/webstore` (Chrome 网上应用店)

**排查方法**：检查报错时的 `tab.url` 是什么。如果是上述任何一种，这是正常行为，无法通过代码解决，只能在发送前判断 URL 并跳过。

#### 3. 竞态条件 (Race Condition)：脚本还没跑起来
这是动态交互中最容易踩的坑。
*   **场景**：
    1.  用户点击按钮。
    2.  Background 代码立即执行 `chrome.tabs.sendMessage`。
    3.  **但是**，目标 Tab 可能刚刚打开，或者刚刚刷新。
    4.  浏览器的 Content Script 注入机制虽然很快，但不是原子的。如果 Background 在 Content Script 完成 `addListener` 注册**之前**就发送了消息，接收端就不存在。
*   **你的代码风险**：如果这段 `sendMessage` 是在用户点击瞬间触发的，而页面正处于加载初期，极易发生此问题。

#### 4. 页面刷新导致的“僵尸”Tab ID
*   **场景**：
    1.  你获取了 `tab.id` (例如 ID=100)。
    2.  用户在 Tab 100 中按下了 F5 刷新。
    3.  **关键机制**：页面刷新时，旧的 Content Script 实例会被销毁，新的实例会重新注入。但是，**Tab ID (100) 保持不变**。
    4.  如果在刷新的这几百毫秒内，Background 尝试向 ID=100 发送消息，旧的监听器已死，新的还没活，就会报错。

#### 5. Content Script 内部报错导致监听器未注册
如果你的 `content-script.js` 顶部有语法错误，或者在 `addListener` 之前执行了报错的代码（例如调用了未定义的变量），脚本执行会中断，`addListener` 永远不会被执行。
*   **检查**：打开目标页面的 F12 控制台，查看是否有红色的 JS 报错，且报错位置在 `content-script.js` 中。

#### 6. Manifest V3 的 Service Worker 休眠问题 (较少见但存在)
虽然主要影响 Background 收不到消息，但在某些极端复杂的异步流程中，如果 Content Script 试图先发消息给 Background 建立握手，而 Background 刚被唤醒，也可能出现时序问题。不过针对你现在的报错（Background 发给 Content），主要是前 5 点原因。

### 三、终极解决方案：构建健壮的通信机制

不要指望 `sendMessage` 一次成功。你需要一套**“检测 - 重试 - 降级”**的机制。

#### 方案 A：发送前的“预检”与 URL 过滤 (基础防御)

在发送消息前，先获取 Tab 详情，排除受限页面。

```javascript
async function safeSendCoverRequest(tabId) {
  try {
    // 1. 获取 Tab 详细信息
    const tab = await chrome.tabs.get(tabId);
    
    // 2. 过滤受限页面 (根据实际需求调整)
    const restrictedPrefixes = ['chrome://', 'chrome-extension://', 'about:', 'file://'];
    if (restrictedPrefixes.some(prefix => tab.url.startsWith(prefix))) {
      console.warn('跳过受限页面:', tab.url);
      return; 
    }

    // 3. 尝试发送消息
    // 注意：这里依然可能报错，因为无法避免刷新时的竞态条件
    const response = await sendMessageWithRetry(tabId, { action: 'getCoverImage' }, 3);
    
    // 处理响应...
    if (response?.error) {
       console.error('业务逻辑错误:', response.error);
       return;
    }
    if (response?.dataUrl) {
       // 执行下载逻辑...
    }

  } catch (err) {
    console.error('发送消息彻底失败:', err);
    // 这里可以触发 UI 提示用户“请刷新页面后重试”
  }
}

// 带重试机制的发送函数
function sendMessageWithRetry(tabId, message, maxRetries = 3) {
  return new Promise((resolve, reject) => {
    let attempts = 0;

    const attempt = () => {
      attempts++;
      chrome.tabs.sendMessage(tabId, message, (response) => {
        if (chrome.runtime.lastError) {
          console.log(`尝试 ${attempts}/${maxRetries} 失败:`, chrome.runtime.lastError.message);
          
          if (attempts < maxRetries) {
            // 等待 200ms 后重试，给 Content Script 启动时间
            setTimeout(attempt, 200); 
          } else {
            reject(new Error('Max retries reached: ' + chrome.runtime.lastError.message));
          }
        } else {
          resolve(response);
        }
      });
    };

    attempt();
  });
}
```

#### 方案 B：主动握手机制 (最稳健，推荐)

不要等用户点击才去联系 Content Script。让 Content Script 在加载完成后，**主动**告诉 Background “我准备好了”。

**1. 修改 content-script.js**:
```javascript
// 页面加载完成后，主动通知 Background
chrome.runtime.sendMessage({ action: 'contentScriptReady', url: window.location.href });

// 原有的监听逻辑保持不变
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.action === 'getCoverImage') {
    onMessage(message, sender, sendResponse);
    return true; 
  }
});

// ... onMessage 函数保持不变 ...
```

**2. 修改 background.js**:
维护一个“就绪列表”。

```javascript
// 存储已就绪的 Tab ID
const readyTabs = new Set();

// 监听 Content Script 的就绪消息
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.action === 'contentScriptReady' && sender.tab) {
    console.log(`Tab ${sender.tab.id} 已就绪`);
    readyTabs.add(sender.tab.id);
    
    // 可选：如果此时正好有排队的下载任务，可以在这里触发
  }
});

// 当用户点击下载时
async function handleDownloadClick(tabId) {
  // 检查是否在就绪列表中
  if (!readyTabs.has(tabId)) {
    console.warn('Content Script 尚未就绪或页面不受支持。尝试最后一次直接发送...');
    // 可以选择在这里调用上面的 retry 逻辑，或者直接提示用户刷新
    alert('页面尚未完全加载，请刷新页面后再试，或正在自动重试...');
    // 触发一次带重试的发送
    return safeSendCoverRequest(tabId); 
  }

  // 如果已就绪，直接发送（成功率极高）
  try {
     const response = await chrome.tabs.sendMessage(tabId, { action: 'getCoverImage' });
     // 处理下载...
  } catch (e) {
     // 即使有列表，也可能因为用户刚刚刷新而失效，移除该 ID
     readyTabs.delete(tabId);
     console.error('发送失败，Tab 可能已刷新', e);
  }
}

// 清理机制：当 Tab 关闭或更新（刷新）时，移除记录
chrome.tabs.onRemoved.addListener((tabId) => readyTabs.delete(tabId));
chrome.tabs.onUpdated.addListener((tabId, changeInfo) => {
  if (changeInfo.status === 'loading') {
    // 页面开始重新加载，旧脚本失效，移除标记
    readyTabs.delete(tabId);
  }
});
```

### 四、针对你代码的具体修正建议

你的 `content-script.js` 逻辑本身有一个小隐患：`getImageAsDataUrl` 看起来是异步的，但你没有处理它可能抛出的异常。如果它内部报错，`sendResponse` 永远不会被调用，虽然这不会导致 "Receiving end does not exist"（因为监听器已经在了），但会导致 Background 一直收不到回复（超时）。

**优化后的 content-script.js:**

```javascript
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.action === 'getCoverImage') {
    onMessage(message, sender, sendResponse);
    return true; // 保持异步
  }
});

async function onMessage(message, sender, sendResponse) {
  console.log('收到消息:', message);
  
  try {
    if (message.action !== 'getCoverImage') {
      sendResponse({ error: 'Unknown action' });
      return;
    }

    const url = getCoverImageUrl();
    if (!url) {
      sendResponse({ error: 'Cover image not found' });
      return;
    }

    // 假设 getImageAsDataUrl 是一个 Promise 或者接受回调
    // 如果是回调风格，确保封装成 Promise 更稳妥，或者像下面这样处理
    getImageAsDataUrl(({ url: dataUrl, code, starName }) => {
      if (!dataUrl) {
         sendResponse({ error: 'Failed to convert image to DataUrl' });
         return;
      }
      
      const filename = `${starName || 'unknown'}/${code || 'no_code'}/${code || 'javbus_cover'}.jpg`;
      sendResponse({ dataUrl, filename });
    });

  } catch (err) {
    console.error('Content script 内部错误:', err);
    // 即使出错也要回复，防止 Background 挂起
    sendResponse({ error: err.message || 'Internal server error in content script' });
  }
}
```

### 五、调试清单 (Checklist)

如果问题依旧，请按此顺序操作：

1.  **打开扩展管理页** (`chrome://extensions`) -> 开启右上角 **"开发者模式"**。
2.  **查看背景页日志**：点击你的扩展卡片上的 "Service Worker" (MV3) 或 "background page" (MV2) 链接，看 Console 里的具体报错。
3.  **打开目标网页** -> F12 开发者工具。
4.  **切换上下文**：在 Console 顶部的下拉框中，选择你的扩展名称（通常在 `top` 下方）。**这一步至关重要**，默认是 `top`，你看不到 Content Script 的日志。
5.  **观察日志**：
    *   有没有看到 `content-script 收到消息`？
        *   **有**：说明连接成功了，错误可能是后续代码逻辑导致的（或者是之前的某次尝试报错）。
        *   **没有**：说明脚本根本没运行。
6.  **检查 Manifest**：复制当前网页 URL，去 `manifest.json` 里用正则测试一下 `matches` 是否匹配。
7.  **强制刷新**：在目标网页按 `Ctrl+F5` 硬刷新，确保加载的是最新版的 Content Script。

### 总结

这个错误**不是**因为你 `return true` 没写好，也不是因为 `sendMessage` 语法错了。
**根本原因**通常是：**在 Background 发起请求的那一刻，目标页面里没有活着的 Content Script 监听器。**

**最佳实践**是采用 **方案 B (主动握手)** 配合 **方案 A (重试机制)**。不要假设 Content Script 随时可用，要把它当作一个可能随时断开或重启的服务来处理。
