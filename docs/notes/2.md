这个错误 `The message port closed before a response was received` 是 Chrome 扩展开发中非常经典的问题。

### 核心原因
在 Manifest V3 的 Content Script 中，当你使用 `chrome.runtime.onMessage.addListener` 并返回 `true`（表示异步响应）时，**必须调用 `sendResponse()` 函数来结束通信**。

在你的代码中，问题出在 `getImageAsDataUrl` 这个函数上：
1. 你调用了 `getImageAsDataUrl(...)`。
2. 你在回调中调用了 `sendResponse`。
3. **但是**，如果 `getImageAsDataUrl` 内部发生错误、或者它根本没有执行回调（例如它是同步返回但没写回调逻辑，或者 Promise 被 reject 但没有 catch），那么 `sendResponse` 永远不会被调用。
4. 一旦 Content Script 的执行上下文销毁（或者消息端口超时），而 Background 还没收到回复，就会报这个错。

此外，还有一个常见的陷阱：**如果 `getImageAsDataUrl` 是基于 Promise 的（现代写法通常如此），而你把它当作回调函数使用，但没有正确处理 Promise 的链式调用，也会导致 `sendResponse` 无法在预期时间内执行。**

---

### 解决方案

建议将 `onMessage` 处理函数重构为 `async/await` 模式，并添加完善的 `try...catch` 错误处理，确保无论成功还是失败，`sendResponse` 都会被调用。

#### 修改后的 `content-script.js`

```javascript
// 监听 background 或 popup 发来的消息
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  // 使用 async IIFE (立即执行函数表达式) 来处理异步逻辑
  (async () => {
    try {
      if (message.action === 'getCoverImage') {
        console.log('content-script 收到消息：', message);
        
        const url = getCoverImageUrl();
        if (!url) {
          // 同步错误：直接返回
          sendResponse({ error: 'Cover image not found' });
          return;
        }

        // 假设 getImageAsDataUrl 返回一个 Promise
        // 如果它原本是回调风格，你需要先把它包装成 Promise (见下方说明)
        const result = await getImageAsDataUrlPromise(url); 
        
        const filename = `${result.starName}/${result.code}/${result.code || 'javbus_cover'}.jpg`;
        
        // 成功：发送数据
        sendResponse({ dataUrl: result.url, filename });
      } else {
        // 未知动作
        sendResponse({ error: 'Unknown action' });
      }
    } catch (err) {
      console.error('Content script processing error:', err);
      // 异常：发送错误信息，防止端口关闭报错
      sendResponse({ error: err.message || 'Internal server error in content script' });
    }
  })();

  // 重要：返回 true 表示我们将异步发送响应
  return true;
});

/**
 * 辅助函数：如果你的 getImageAsDataUrl 是旧式的回调风格，请将其包装为 Promise
 * 如果它已经是 Promise 风格，可以直接使用，无需此包装。
 */
function getImageAsDataUrlPromise(url) {
  return new Promise((resolve, reject) => {
    // 调用原有的回调风格函数
    getImageAsDataUrl({ url }, (result) => {
      if (result && result.url) {
        resolve(result);
      } else {
        reject(new Error('Failed to get image data'));
      }
    });
    
    // 【可选】添加超时保护，防止回调永远不触发导致端口挂起
    // setTimeout(() => reject(new Error('Get image timeout')), 5000); 
  });
}
```

### 关键修改点解析

1.  **包裹在 `try...catch` 中**：
    这是最关键的一步。原来的代码如果 `getImageAsDataUrl` 抛出异常或者逻辑走不通，`sendResponse` 就不会执行。现在无论发生什么，`catch` 块都会捕获错误并调用 `sendResponse({ error: ... })`，从而正确关闭消息端口。

2.  **使用 `async/await`**：
    这让代码逻辑更清晰。我们不再依赖深层嵌套的回调，而是等待 `getImageAsDataUrl` 完成。

3.  **Promise 包装（如果需要）**：
    看你原本的代码 `getImageAsDataUrl(({ url, code, starName }) => { ... })`，这看起来像是一个接受配置对象和回调函数的传统 API。
    *   如果 `getImageAsDataUrl` **没有** 返回 Promise，你必须像上面代码中的 `getImageAsDataUrlPromise` 那样，手动把它包一层 `new Promise`。
    *   如果在包装函数里，原函数因为某些原因（比如参数不对）**完全没有调用回调函数**，Promise 就会一直 pending。为了防止这种情况导致报错，建议在 Promise 里加一个 `setTimeout` 作为超时熔断（代码中已注释示例）。

4.  **确保 `sendResponse` 必达**：
    检查你的 `getCoverImageUrl()` 和 `getImageAsDataUrl` 实现。确保在所有分支（包括图片加载失败、DOM 未找到等）下，最终都能触发 `resolve` 或 `reject`，进而触发 `sendResponse`。

### 额外检查清单

如果修改后仍然报错，请检查以下几点：

*   **Content Script 是否被意外卸载？**
    如果你在等待图片加载的过程中，用户刷新了页面或关闭了标签页，Content Script 会被销毁，端口自然关闭。这是正常现象，但通常不会在控制台报这个特定的错，除非 Background 还在尝试发消息。
*   **`getImageAsDataUrl` 的实现逻辑**：
    请确认该函数内部是否真的会调用回调。例如，如果它是通过 `fetch` 获取图片，是否处理了 `catch`？如果 `fetch` 失败且没有 catch，回调就不会执行。
*   **Manifest 权限**：
    虽然你的 manifest 看起来没问题，但确保 `content_scripts` 确实注入到了目标页面。如果脚本没运行，当然收不到消息（但这通常会报 "Could not establish connection" 而不是 "port closed"）。

采用上面的 `try...catch` + `async/await` 模式通常能解决 99% 的此类问题。
